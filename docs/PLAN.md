## Problem Statement

Build a wearable haptic navigation belt that fuses computer vision, voice AI, and IMU sensing to give directional guidance through vibration. The product must work end-to-end within 2 days as a functional prototype.

## What Makes This Cool

The novel core loop: **camera sees the world → AI reasons about it → body feels the answer.** Not a screen, not a speaker — a motor that taps you on the left hip and you turn left. Combined with a voice agent ("find the exit"), it becomes a tangible AI assistant you wear.

## Locked-In Premises

1. **Drop DoorDash-style AI assistant.** Voice agent is in scope but scoped to belt-relevant intents: navigation queries, fall-response, status. Not a general chatbot.
2. **Webapp = demo/debug console, not product UI.** Live camera feed, YOLO boxes, IMU trace, current haptic pattern. No auth, no routing, no polish beyond readable.
3. **USB-CDC for ESP↔Pi.** Serial with a framed JSON protocol. No WiFi/BT for this link.
4. **PTT > wake-word.** Button A on M5 toggles listening. Reliable in noisy venues.
5. **Primary YOLO runs on Pi CPU** (YOLOv8n, 320×320, ~5 FPS). Laptop is the fallback if Pi is too slow.
6. **Distance ToF is first-class input**, not a nice-to-have. It catches obstacles YOLO misses and runs at 50Hz.

## Interface Contracts

These are the source of truth for parallel work. Lock them now, no renegotiation after Day 1 noon.

### 1. Serial Protocol (Pi ↔ M5, USB-CDC, 115200 baud)

**Framing:** Newline-delimited JSON. One message per line. UTF-8. No partial frames.

**Pi → M5 messages:**
```json
{"t":"haptic","dir":"L","intensity":180,"pattern":"pulse","duration_ms":500}
// dir: F|B|L|R|ALL, intensity: 0-255, pattern: solid|pulse|ramp|sos
{"t":"display","line1":"Navigating...","line2":"3 obstacles"}
{"t":"ping","seq":42}
```

**M5 → Pi messages:**
```json
{"t":"imu","ax":0.1,"ay":9.8,"az":0.2,"gx":0,"gy":0,"gz":0,"ts":1700000000}
{"t":"ptt","state":"down"}  // or "up"
{"t":"audio","b64":"...base64 PCM16 16kHz frame..."}  // 20ms frames
{"t":"fall","severity":"hard","az_peak":-28.4}
{"t":"modulino_imu","ax":...}   // from Modulino Movement
{"t":"distance","mm":420}
{"t":"pong","seq":42}
{"t":"hello","fw":"1.0.0","caps":["imu","mic","haptic","display"]}
```

**Heartbeat:** Pi sends `ping` every 1s. If no `pong` for 3s → reset serial.

### 2. I2C Address Map (M5 I2C bus, Grove port, 3.3V/400kHz)

| Device | Address | Notes |
|--------|---------|-------|
| Modulino Vibro Front | 0x5A | Rotate address straps per board |
| Modulino Vibro Back  | 0x5B | |
| Modulino Vibro Left  | 0x5C | |
| Modulino Vibro Right | 0x5D | |
| Modulino Movement    | 0x6A | BMI270 default |
| Modulino Distance    | 0x29 | VL53L4CD default |

**Day 0 verification:** Ayan runs an I2C scanner sketch and confirms all 6 devices enumerate before anyone else codes against this map.

### 3. WebSocket Schema (Pi → Webapp, port 8000, `/ws`)

Server broadcasts at ~10Hz. Types:
```json
{"t":"frame","jpeg_b64":"...","w":320,"h":320}
{"t":"detections","boxes":[{"cls":"chair","conf":0.82,"x":0.3,"y":0.5,"w":0.1,"h":0.2}]}
{"t":"imu","source":"m5","ax":...,"ay":...,"az":...}
{"t":"distance","mm":420}
{"t":"haptic","dir":"L","intensity":180}
{"t":"voice","role":"user|assistant","text":"find the exit"}
{"t":"fall","severity":"hard"}
{"t":"health","pi_cpu":0.42,"serial_ok":true,"yolo_fps":4.7}
```

### 4. Navigation Fusion Logic (the core algorithm)

Inputs every tick (target 10Hz):
- YOLO boxes with class + normalized xywh (latest frame)
- Distance ToF reading (mm)
- Voice intent state (IDLE | SEEKING(target_class) | EMERGENCY)
- IMU fall flag

Priority (highest first):
1. **Fall detected** → `haptic:ALL,sos,1500ms` + voice "are you okay?" + webapp banner
2. **Distance < 800mm** → `haptic:F,pulse,intensity=255-inverse(dist)` (closer = stronger)
3. **YOLO: obstacle in path** (box center in middle third, dist proxy via bbox height) → haptic on opposite side to suggest turn
4. **Voice SEEKING mode** → if target class visible, haptic points toward it; else gentle `ALL,ramp` "searching"
5. **Idle** → no haptic

Rate limit: no more than one haptic event per 250ms per direction. No more than 3 seconds sustained vibration (safety).

## Phased Build Plan (2 days, 4 phases)

Each phase is ~4–6h. Buffer for sleep, meals, and chaos is built in.

### Phase 1 — Scaffolding (Day 1, 0–5h)

**Goal:** Every person has their subsystem saying "hello world" independently. No integration yet.

**Exit criteria (all 4 must be true):**
- M5 boots, reads its IMU, and prints `{"t":"hello",...}` over USB.
- Pi boots, Python env ready, camera returns one JPEG to disk.
- Modulino bus I2C-scans clean; Vibro 0x5A buzzes on demand.
- Belt housing v1 printed; motor mount brackets fit Modulino boards.

### Phase 2 — Per-Subsystem Core (Day 1, 5–12h)

**Goal:** Each subsystem fully functional in isolation, behind its contract.

**Exit criteria:**
- M5 streams IMU at 50Hz, detects simulated fall, handles `haptic` commands, streams mic audio frames.
- Haptic library: `buzz(dir, intensity, pattern, duration)` works for all 4 Vibros.
- Pi: YOLO runs on sample JPEGs, produces bounding boxes ≥3 FPS. Voice round-trip (PTT → text → LLM → TTS audio) works with keyboard-simulated PTT.
- Webapp: FastAPI server runs, WS accepts fake messages, renders camera feed + detections + haptic indicator + IMU trace.
- CAD v2: final belt, motor placements ergonomic, Pi pouch fits with power brick.

### Phase 3 — Integration (Day 2, 0–6h)

**Goal:** Wire subsystems together. Real data flowing end-to-end.

**Exit criteria:**
- Pi ↔ M5 serial link stable for 10 minutes, no dropped frames.
- PTT button on M5 → mic audio frames → Pi → OpenAI Realtime → TTS → Pi speaker (or M5 speaker).
- YOLO on live camera → navigation fusion → haptic command → real motor buzzes.
- Distance ToF reading flows M5 → Pi → fusion → haptic override when < 800mm.
- Fall detection on M5 → Pi alert → webapp banner + voice prompt.
- Webapp shows live everything.

### Phase 4 — Polish & Hardening (Day 2, 6–10h)

**Goal:** Make it reliable. Not pretty, reliable.

**Tasks:**
- 30-minute burn-in: belt on a person walking around, no crashes, no serial resets, no disconnects.
- Calibrate: fall threshold (no false positives from bending down), distance threshold (800mm may need tuning), motor intensities (top value shouldn't hurt).
- Voice agent: tighten system prompt to belt-context-only. Add two test intents: "what do you see" and "find the [object]".
- Webapp: one-click "start demo" button that brings all services up.
- Fallback flips ready: laptop+webcam swap-in script, local-LLM flip, and direct-motor-drive firmware branch.

### Vaibhav — Firmware Lead (M5 + Protocol)

**Tooling:** PlatformIO + Arduino framework + M5StickCPlus library.

**Phase 1:**
- Bring up M5 StickC Plus: LCD hello world, MPU6886 read loop, PDM mic sample capture.
- Define and implement the JSON serial protocol (send/receive). Add `hello`, `ping`, `pong`.

**Phase 2:**
- Finalize serial state machine: robust parser (rolling JSON line buffer), malformed-line recovery.
- Mic streaming: 16kHz PCM16, 20ms frames, base64, emit `audio` messages only between PTT down/up.
- Fall detection: accel magnitude > 2.5g for <150ms followed by <0.3g free-fall signature → emit `fall`. Tunable thresholds via `#define`.
- Display handler: render `display` message text on LCD.

**Phase 3:**
- Lead serial integration with Sid. Latency budget: command-to-motor < 50ms.
- Add heartbeat watchdog: reset I2C bus if Modulino calls fail 3× consecutive.

**Phase 4:**
- Burn-in + tuning. Owns the M5 firmware binary that gets flashed for demo.
- Emergency firmware branch: **direct-motor-drive** — if Modulino I2C fails at integration, drive motors off 4 GPIOs with MOSFET breakout + PWM. Keep this branch stubbed and ready.

### Ayan — Firmware/Software (Modulino + Haptic Library + Hardware Integration)

**Tooling:** PlatformIO (shares M5 codebase), multimeter, scope if available.

**Phase 1:**

**Phase 2:**
- Haptic library in C++: `Haptic::buzz(Direction dir, uint8_t intensity, Pattern pat, uint16_t duration_ms)`.
  - Non-blocking, timer-driven, supports overlapping calls (queue + priority).
  - Patterns: `SOLID`, `PULSE` (200ms on / 200ms off), `RAMP` (0→255 over duration), `SOS` (···—-—-···).
- Modulino Movement driver: emit `modulino_imu` at 20Hz.
- Modulino Distance driver: emit `distance` at 50Hz.
- Wire harness v1 — daisy-chain Qwiic cables around the belt, strain relief, label connectors.

**Phase 3:**
- Pair with Vaibhav to connect `haptic` serial messages → `Haptic::buzz` calls.
- Hardware integration with Sam: mount electronics, route cables, secure to belt, verify wearability.

**Phase 4:**
- Cable tension test: wearer sits, stands, turns — nothing unseats.
- Document hardware pinout and wiring diagram for the final writeup.
- Own the emergency MOSFET-direct-drive breakout — prebuild it, leave it unplugged, ready to swap.

**Files Ayan owns:** `firmware/m5/src/haptic/*`, `firmware/m5/src/modulino/*`, `docs/wiring-diagram.pdf`, `hardware/bom.md`.

## Success Criteria

Product works end-to-end when:

1. Wearer walks forward, camera sees an obstacle, front motor buzzes, wearer turns away, motor stops.
2. Wearer holds PTT, says "what do you see," voice replies with a sentence describing the current frame.
3. Wearer falls (or simulates): belt buzzes SOS, webapp shows banner, voice asks if they're okay.
4. Webapp is open on a laptop showing live camera, boxes, IMU, haptic state, voice transcript.
5. System runs 15 minutes continuously without intervention.
